# FundTogether 众筹平台技术规范

## 一、 文档前言

### 1.1 文档目的与范围
本文档作为 FundTogether 众筹平台的权威技术参考，旨在将项目业务需求（基于 `README.md`）落地为详细的技术实现方案。指导开发人员完成全流程的架构设计、功能开发、接口定义与项目部署。
适用范围覆盖系统的整个开发周期，包括用户认证、项目管理、订单支付、后台运营以及平台实时消息等所有业务环节。

### 1.2 术语与交叉引用定义
为确保技术架构与业务需求无缝衔接，本文档与需求说明中的术语严格对齐：
- **游客 (Visitor)**：未认证用户，享有只读接口权限及注册入口权限。
- **普通支持者 (Supporter)**：通过认证机制的登录用户，具备投资支持、评论互动权限。
- **项目发起人 (Sponsor)**：具备项目创建及全生命周期维护权限的特殊用户角色。
- **系统管理员 (Admin)**：最高后台操作权限拥有者，执行全站系统风控及监管。
- **订单状态机 (Order State Machine)**：映射需求中的“待支付”、“已支付”、“已取消”状态。
- **项目状态机 (Project State Machine)**：映射需求中的“待审核”、“筹款中”、“已驳回”、“已取消”、“已下架”状态。

## 二、 项目概述

FundTogether 是一套全栈开发的在线众筹网站。从技术落地角度来看，系统旨在构建一条闭环数据流，贯穿**用户注册认证 -> 发起人创建项目 -> 管理员风控审核 -> 用户下单与支付 -> 数据统计与实时消息推送**。

**对应 README 需求映射：**
- **项目简介与角色**：完美对应 `README.md` 中定义的四类角色，后端将在 SpringSecurity 中配置与之严格对齐的 RBAC（基于角色的访问控制）权限校验策略。

## 三、 技术架构选型

根据业务需求中的多端适配、响应速度与交易一致性要求，整体采用经典的前后端分离架构，通过容器化技术实现环境隔离与快速部署。

| 技术领域 | 技术选型 | 选型理由（技术保障措施） |
| :--- | :--- | :--- |
| **前端核心框架** | Vue3 + Vite | Composition API 提供优异的逻辑复用，Vite 保障开发环境极速热更新及构建效率。 |
| **UI组件与适配** | Element Plus + CSS3 | 响应式栅格系统和 Flex 布局直接实现跨端适配，满足 `README` 中对移动端的展示要求。 |
| **可视化图表** | ECharts | 支持按需求绘制分类筹款柱状图、趋势折线图及饼图，实现后台核心数据看板。 |
| **后端核心框架** | SpringBoot 3.x | 简化约定配置，提供高可靠的 RESTful 服务并内嵌 Tomcat 引擎。 |
| **持久层框架** | MyBatis-Plus | 基于 MyBatis 增强，自带分页插件满足需求列表翻页要求，逻辑删除特性保障数据安全。 |
| **权限与安全** | SpringSecurity + JWT | 无状态令牌机制减轻服务器负担，提供精准的 RBAC 接口级拦截（防越权操作）。 |
| **关系型数据库** | MySQL 8.0 | InnoDB 引擎的强事务 ACID 特性保障支付订单与项目筹款金额的强一致性。 |
| **缓存与锁机制** | Redis 7.x | 作为多级缓存提升项目列表/详情等高频接口响应，同时通过分布式锁防止订单超卖、重复提交。 |
| **网关与代理** | Nginx | 负责静态资源托管、请求转发负载均衡，并通过配置请求频次实现接口防刷限流。 |
| **全双工通信** | WebSocket | 建立服务端与多端客户端的长连接，满足平台公告、动态更新的实时消息推送需求。 |
| **容器化与部署** | Docker + Compose | 屏蔽不同环境带来的开发部署差异，实现后端服务与中间件的一键编排。 |

## 四、 核心业务模块技术实现方案

本章节针对 `README.md` 提及的 7 大核心功能模块提供具体实现路径。

### 4.1 认证与用户管理
- **业务需求**：多渠道注册、Token状态管理、实名绑定。
- **技术实现**：
  1. 采用 `Bcrypt` 算法进行密码单向哈希加密存储。
  2. 登录成功后，后端生成带过期时间（如2小时）的 `JWT` 并下发，客户端统一在 Header 中以 `Authorization: Bearer <token>` 携带请求。
  3. Redis 配合缓存手机/邮箱验证码（Key格式：`sms:code:{phone}`），并设定5分钟TTL及防刷限制。
  4. 头像与实名认证图片等非结构化数据通过独立文件上传接口处理，存储于服务器指定目录或对接 OSS。

### 4.2 项目浏览与交互
- **业务需求**：大厅展示、多维检索、数据折线图。
- **技术实现**：
  1. 引入 MyBatis-Plus `IPage` 插件完成复杂多条件查询（分类、状态、排序规则）。
  2. 对访问频次极高的“热门推荐项目”采取 Redis 缓存（设置合理的TTL，并定时刷新）。
  3. 项目详情页涉及富文本和多图加载，数据库端使用 `TEXT` 或 `LONGTEXT` 存储，前端使用 `v-html` 进行XSS清洗后渲染。

### 4.3 项目支持与交易闭环
- **业务需求**：生成支持订单、对接三方支付渠道、保证状态强一致性。
- **技术实现**：
  1. **事务保障**：订单创建、金额计算及项目进度更新必须放置在同一个被 `@Transactional(rollbackFor = Exception.class)` 标注的方法内，保证操作的原子性。
  2. **防重复提交**：采用 Redis 分布式锁（如 `order:lock:{userId}:{projectId}`），锁释放前拒绝同参数的二次请求。
  3. **支付回调验签**：调用支付宝/微信沙箱环境，并在回调接口 (`/api/v1/pay/callback`) 进行严格的 RSA 签名验证，校验金额、订单号防篡改。

### 4.4 项目发起与管理
- **业务需求**：表单与媒体资源提交、项目状态机管理。
- **技术实现**：
  1. 建立项目状态枚举类：`PENDING_REVIEW`（待审核）、`FUNDING`（筹款中）、`REJECTED`（已驳回）、`CANCELED`（已取消）、`OFFLINE`（已下架）、`SUCCESS`（已完成）。
  2. 通过后端逻辑强制校验状态流转的合法性（如：“已下架”项目不能被重新编辑）。
  3. 针对音视频等大体积媒体，上传模块开启文件格式拦截（仅允许 mp4/jpg/png）并限制最大体积（如视频100MB），必要时采用分片上传。

### 4.5 后台监管与运营
- **业务需求**：全站项目风控、多维度统计报表导出。
- **技术实现**：
  1. **数据看板**：编写复杂聚合 SQL (`GROUP BY` 与聚合函数) 统计各分类项目数据及趋势，提供给前端 ECharts 渲染。
  2. **报表导出**：引入 `Alibaba EasyExcel` 组件，利用低内存占用模式将数据流式写入 `.xlsx` 文件供管理员下载。

### 4.6 消息与通知机制
- **业务需求**：实时项目动态与全站公告推送。
- **技术实现**：
  1. 后端集成 `spring-boot-starter-websocket`，鉴权拦截器允许合法 Token 建立 ws 握手。
  2. 按连接会话维护在线用户池，发生业务动作（如项目状态更新）时触发事件机制，服务端主动 `push` 结构化 JSON 消息体。

### 4.7 跨端适配与体验
- **业务需求**：PC 与移动端自适应。
- **技术实现**：
  1. 页面采用 CSS3 媒体查询（`@media`）与 Element Plus 响应式特性。
  2. 不区分 PC/H5 接口，后端提供统一标准的 RESTful JSON 接口集。

## 五、 API 设计规范

1. **RESTful 标准**
   - URL一律使用小写名词复数结构，通过 HTTP Method 表意。
   - 示例：`GET /api/v1/projects` (列表)；`POST /api/v1/sponsor/project` (创建)；`PUT /api/v1/sponsor/project/{id}` (更新)；`DELETE /api/v1/sponsor/project/{id}` (删除)。
2. **统一响应协议**
   ```json
   {
     "code": 200,          // 业务状态码，200 为成功
     "msg": "操作成功",      // 对用户友好的提示信息
     "data": { ... }       // 实际响应数据，列表或对象
   }
   ```
3. **接口安全控制**
   - 后端基于 SpringSecurity 配置拦截链。公开接口放行，其余接口强制拦截校验 Header 侧的 `Authorization: Bearer <Token>` 签名及有效期。

## 六、 开发规范

为确保 Vibe Coding 开发过程的质量可控，制定以下硬性开发规范。

### 6.1 代码风格指南
1. **Java/后端**：严格遵循《阿里巴巴 Java 开发手册》。
   - 缩进：4 空格。
   - 命名：类 `UpperCamelCase`，方法与变量 `lowerCamelCase`，常量大写下划线 `UPPER_SNAKE_CASE`。
   - 集合操作：避免在循环体内执行 SQL 操作，防止 N+1 查询问题。
2. **Vue/前端**：遵循 Vue3 官方 Style Guide。
   - 缩进：2 空格（由 ESLint + Prettier 约束）。
   - 组件：文件名使用 PascalCase（如 `ProjectDetail.vue`）。优先使用 `<script setup>` 与 `const`。

### 6.2 分支管理策略
采用精简版 Git Flow 策略：
- `main`：生产稳定分支，对应每次发布的 Tag，禁止直接 Push。
- `develop`：开发环境主干分支，日常合码基线。
- `feature/{功能名}`：功能开发分支，由 `develop` 检出，开发完毕通过 MR 合并。
- `hotfix/{问题单号}`：紧急 Bug 修复分支，由 `main` 检出。

### 6.3 代码审查流程 (Code Review)
1. 开发者完成本地开发并跑通自测后，发起至 `develop` 或 `main` 的 Pull Request/Merge Request。
2. 必须由至少 1 名项目维护者进行 Code Review。
3. **审查维度**：
   - 接口契约是否与本规范和需求定义完全一致。
   - 复杂 SQL 性能及表索引设计合理性。
   - 资源（流、连接锁）是否妥善处理关闭。
   - 必要的边界异常 `try-catch` 逻辑与明确的系统提示。

### 6.4 测试覆盖率标准
1. **单元测试**：针对高危/核心业务逻辑（如订单金额计算、Token 拦截生成、分布式并发扣减机制），强制编写 JUnit 单元测试，保证 Service 层核心逻辑覆盖率 > 80%。
2. **接口测试**：集成 Knife4j 生成接口文档，开发交付前需确保 Postman 冒烟测试全覆盖。

## 七、 非功能性需求保障与部署指南

### 7.1 非功能性技术保障
1. **安全保障**：XSS 过滤与 SQL 预编译处理；密码禁止明文入库；日志禁止打印敏感信息（身份证、密码脱敏）。
2. **性能保障**：数据库单表数据超限前采用索引优化；对于热点读操作全面引入 Redis 缓存；配置 Nginx 接口限流拦截非法大流量访问。

### 7.2 部署指南
1. **Docker 容器化**：系统基于 Docker 运行环境，包含应用服务、MySQL 8.0 及 Redis 7 镜像。
2. **编排文件**：根目录下提供 `docker-compose.yml`，执行 `docker-compose up -d` 即可拉起完整全栈环境。
3. **环境隔离**：配置文件通过 `application-dev.yml`、`application-test.yml`、`application-prod.yml` 隔离不同环境。数据库账密必须通过环境变量注入。
